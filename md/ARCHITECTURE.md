# 🏗 Архитектура CatDock

CatDock — это сложная распределенная система, объединяющая Telegram-бота, веб-интерфейс и управление Docker-контейнерами.

## 🧩 Основные компоненты

### 1. Core (Ядро)
*   **Язык:** Python 3.10+
*   **Фреймворк:** Aiogram 3.x (Бот) + FastAPI (API).
*   **Роль:** Обработка команд, управление базой данных, бизнес-логика.
*   **Запуск:** Работает как системный сервис через PM2 (`pm2 start bot.py`).

### 2. Database (База данных)
*   **СУБД:** PostgreSQL 15.
*   **Драйвер:** `asyncpg` (для асинхронной работы бота), `psycopg2` (для миграций).
*   **Миграции:** Alembic (управление версиями схемы БД).
*   **Развертывание:** В Docker-контейнере `catdock_db`.
*   **Роль:** Хранение пользователей, контейнеров, балансов, тикетов.

### 3. Frontend (Веб-панель)
*   **Стек:** HTML5, Tailwind CSS, Alpine.js.
*   **Тип:** SPA-like (Single Page Application feel), но без сборки (No-Build).
*   **Роль:** Интерфейс для пользователя. Общается с ядром через REST API.
*   **Хостинг:** Статика отдается через Nginx.

### 4. Docker Orchestrator (Управление контейнерами)
*   **Библиотека:** `asyncssh` (для удаленных серверов) + Docker API.
*   **Роль:** Создание, удаление, старт/стоп юзерботов на разных серверах (нодах).
*   **Изоляция:** Каждый юзербот — это отдельный Docker-контейнер с лимитами CPU/RAM.

### 5. Nginx (Веб-сервер)
*   **Роль:**
    1.  Раздача статики (HTML/CSS/JS) из `/var/www/CatDock`.
    2.  Reverse Proxy для API (`/api/` -> `localhost:8082`).
    3.  SSL терминация (HTTPS).

---

## 🔄 Поток данных (Data Flow)

1.  **Пользователь** заходит на `catdock.catdock.io`.
2.  **Nginx** отдает ему `index.html`.
3.  **JS (Alpine)** делает запрос на `/api/v1/auth/...`.
4.  **Nginx** проксирует запрос на порт `8082` (FastAPI).
5.  **FastAPI** проверяет токен в **PostgreSQL**.
6.  Если пользователь хочет перезагрузить контейнер:
    *   **FastAPI** вызывает функцию в `utils.docker`.
    *   **Utils** подключается по SSH к нужному серверу (например, `de-2`).
    *   **SSH** выполняет `docker restart ...`.
